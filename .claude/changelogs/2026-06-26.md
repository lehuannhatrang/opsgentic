## Prometheus metrics tool for the RCA/context agent

Added Prometheus as a third read-only MCP server so the RCA/context enrichment
agent can query metrics and decide query parameters (PromQL, time range, step,
labels) itself.

- `mcp-config/servers.yaml`: new `prometheus` server (streamable_http,
  `${PROMETHEUS_MCP_URL}`, no auth).
- `deploy/manifests/mcp/prometheus-deployment.yaml` + `prometheus-service.yaml`:
  `pab1it0/prometheus-mcp-server` on :8083 (NodePort 31083), HTTP transport,
  `PROMETHEUS_URL` from configmap, no ServiceAccount (HTTP-only to Prometheus).
- `deploy/manifests/kustomization.yaml`: registered the two new manifests.
- `deploy/manifests/configmap.yaml`: `PROMETHEUS_MCP_URL` and `PROMETHEUS_URL`
  (`http://kube-prometheus-stack-prometheus.prometheus.svc:9090`).
- `src/opsgentic/mcp/context.py`: `_gather_async` now includes the `prometheus`
  server alongside `kubernetes`.
- `deploy/manifests/agent-skills/sre.md`: guidance on using metrics to
  corroborate hypotheses (tight time range, instant-first, coarse step).
- Spec: `docs/superpowers/specs/2026-06-26-prometheus-metrics-tool-design.md`.

Verify after deploy with `python -m opsgentic.mcp.diagnose` (confirms transport,
path, and exact tool names per server version).

## Tool-call audit trail (debug what the diagnostics agent ran)

The context/RCA agent now records every tool invocation so a run can be audited
end to end (which tool, the chosen params — PromQL/time range/labels — and the
result). Applies to all tools (k8s/github/prometheus), not just Prometheus.

- `src/opsgentic/mcp/context.py`: new `_summarize_tool_calls` pairs each
  `AIMessage.tool_calls` (name + args) with its `ToolMessage` result (truncated
  to 2000 chars). Results are logged at INFO and embedded in
  `context_data.tool_calls`. `tools_used` is now derived from that list.
- `context_data.tool_calls` flows to the API `/runs/{thread_id}` automatically
  via `runner.snapshot` (which already returns full state minus `messages`).
- `console/public/index.html`: run detail renders a collapsible "Tool calls"
  section (args + truncated result per call). Requires a console image rebuild
  (kustomization `opsgentic-console` tag) to take effect.
- Each tool call now records its origin MCP server. `_gather_async` fetches tools
  per server (`get_tools(server_name=...)`) to build a name->server map, and
  `_summarize_tool_calls` tags every call with `server`
  (kubernetes/prometheus/github). The console shows it as a colored group chip in
  each tool-call summary; the INFO log prefixes the server, e.g.
  `context tool call: [prometheus] execute_query ...`.

## Flatten Prometheus tool output to value-first text

The pab1it0 prometheus-mcp result is verbose (content-block wrapper, escaped
JSON, noisy labels, a Prometheus UI `links` block) — it wastes agent context and
is hard to audit, and range queries can blow up / get truncated. The values were
present (langchain-core 1.x text blocks reach the agent), just buried.

- `src/opsgentic/mcp/context.py`: `_wrap_prometheus_tool` wraps every
  `prometheus` server tool so its output is flattened before it reaches the agent
  and the audit trail. `_flatten_prometheus_result` renders vector results as
  `labels = value` lines and matrix (range) results as per-series
  `last/min/max/n`, dropping `links` and noisy labels (`_PROM_DROP_LABELS`).
  Unexpected shapes (list_metrics, metadata, parse failure) are returned verbatim
  via `_extract_text`, so no data is ever lost. k8s/github tools are untouched.
- `deploy/manifests/agent-skills/sre.md`: instruct the agent to cite concrete
  numeric metric values as evidence, not a Prometheus UI link.

## PR comment agent (human-in-the-loop on remediation PRs)

The agent can now respond to human comments on an opsgentic PR: it re-investigates
(PR diff, original alert/checkpoint context, live Kubernetes / Prometheus / logs)
and replies. A question gets an evidence-backed answer; a suggested change is
verified first — if correct it is committed to the PR branch, if not the agent
replies with a reasoned rebuttal. GitHub only; PR conversation comments only.

Spec: docs/superpowers/specs/2026-06-26-pr-comment-agent-design.md

- `triggers/github.py` (new): HMAC `verify_signature`, `parse_comment_event`,
  `mentions_agent`, `is_self`, `should_respond`. Listen rule for flat comments:
  respond on @mention OR when the most recent prior comment was the agent's
  (detected via the `<!-- opsgentic-agent -->` footer / App slug / bot login).
- `conversation/responder.py` (new): read-only ReAct agent (k8s + prometheus +
  github, plus pods_log) with structured `PRResponse {kind, reply, edits}`;
  `conversation/context.py` (new): assembles run/checkpoint state (via
  `runs.get_by_pr_url`) or PR body, recent comments, and the base...branch diff.
- `mcp/agent_tools.py` (new, refactor): shared `load_tools(include, deny)` +
  `summarize_tool_calls` + `log_tool_calls` + Prometheus flattening, extracted
  from `mcp/context.py` (now a thin consumer). Reused by the responder.
- `gitops/remediator.py`: extracted `edits_to_ops` (shared edit->ops conversion).
- `gitops/pr.py`: `get_pr`, `list_pr_comments`, `post_pr_comment`, `AGENT_FOOTER`.
- `runs.py`: `get_by_pr_url` + `opsgentic_pr_events` dedup table
  (`is_comment_processed` / `mark_comment_processed`).
- `runner.py`: `handle_comment_and_track` + `enqueue_comment`; `tasks.py`:
  `handle_pr_comment` task; `main.py`: `POST /webhook/github` (verify, filter,
  dedup, enqueue).
- `config.py`: `GITHUB_WEBHOOK_SECRET`, `GITHUB_AGENT_HANDLE`,
  `PR_RESPONDER_MAX_COMMENTS`.
- `agent-skills/pr-responder.md` (new, composed with `sre`).
- Deploy: `GITHUB_WEBHOOK_SECRET` (secret), `GITHUB_AGENT_HANDLE` (configmap),
  and `webhook/ingress-tailscale.yaml` — Tailscale Funnel for public HTTPS to the
  private cluster (operator + ACL prereqs documented in the manifest).

Verified: py_compile all modules; unit tests for signature/parse/listen-rule and
edits_to_ops/PRResponse/flatten; full app import + route/task wiring; `kubectl
kustomize` build.

## Console: show PR-comment runs + reduce reply truncation

- PR-comment interactions are now recorded as run rows so they appear in the
  console. `runs.upsert_comment_run` inserts an `opsgentic_runs` row keyed
  `prcomment-<comment_id>` with a `github-comment` summary (title, repo, kind,
  author, the human comment, the agent reply, and the tool-call audit trail).
  `runs.get_by_pr_url` now excludes `prcomment-*` rows so PR-context recovery
  still finds the original alert run (not a later comment run sharing the pr_url).
- `conversation/responder.respond` returns `(PRResponse, tool_calls)`;
  `runner.handle_comment_and_track` records the run via `_record_comment_run`
  (status `applied` when an edit was committed, else `completed`).
- `console/public/index.html`: `renderRun` routes `source==github-comment` to a
  new `renderCommentRun` (comment + reply + kind + tool calls); the tool-call
  rendering is factored into a shared `toolCallsHtml()` reused by both run types.
- Reply truncation mitigation: `agent-skills/pr-responder.md` now tells the agent
  to be economical with tool output (esp. logs — fetch tail/recent only, never
  full dumps) and to write a complete, self-contained reply rather than promising
  an analysis it then omits. Large tool outputs were squeezing the output token
  budget, truncating the final structured reply.
