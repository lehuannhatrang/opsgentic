# 2026-06-26

## Prometheus metrics tool for the RCA/context agent

Added Prometheus as a third read-only MCP server so the RCA/context enrichment
agent can query metrics and decide query parameters (PromQL, time range, step,
labels) itself.

- `mcp-config/servers.yaml`: new `prometheus` server (streamable_http,
  `${PROMETHEUS_MCP_URL}`, no auth).
- `deploy/manifests/mcp/prometheus-deployment.yaml` + `prometheus-service.yaml`:
  `pab1it0/prometheus-mcp-server` on :8083 (NodePort 31083), HTTP transport,
  `PROMETHEUS_URL` from configmap, no ServiceAccount (HTTP-only to Prometheus).
- `deploy/manifests/kustomization.yaml`: registered the two new manifests.
- `deploy/manifests/configmap.yaml`: `PROMETHEUS_MCP_URL` and `PROMETHEUS_URL`
  (`http://kube-prometheus-stack-prometheus.prometheus.svc:9090`).
- `src/opsgentic/mcp/context.py`: `_gather_async` now includes the `prometheus`
  server alongside `kubernetes`.
- `deploy/manifests/agent-skills/sre.md`: guidance on using metrics to
  corroborate hypotheses (tight time range, instant-first, coarse step).
- Spec: `docs/superpowers/specs/2026-06-26-prometheus-metrics-tool-design.md`.

Verify after deploy with `python -m opsgentic.mcp.diagnose` (confirms transport,
path, and exact tool names per server version).

## Tool-call audit trail (debug what the diagnostics agent ran)

The context/RCA agent now records every tool invocation so a run can be audited
end to end (which tool, the chosen params â€” PromQL/time range/labels â€” and the
result). Applies to all tools (k8s/github/prometheus), not just Prometheus.

- `src/opsgentic/mcp/context.py`: new `_summarize_tool_calls` pairs each
  `AIMessage.tool_calls` (name + args) with its `ToolMessage` result (truncated
  to 2000 chars). Results are logged at INFO and embedded in
  `context_data.tool_calls`. `tools_used` is now derived from that list.
- `context_data.tool_calls` flows to the API `/runs/{thread_id}` automatically
  via `runner.snapshot` (which already returns full state minus `messages`).
- `console/public/index.html`: run detail renders a collapsible "Tool calls"
  section (args + truncated result per call). Requires a console image rebuild
  (kustomization `opsgentic-console` tag) to take effect.
- Each tool call now records its origin MCP server. `_gather_async` fetches tools
  per server (`get_tools(server_name=...)`) to build a name->server map, and
  `_summarize_tool_calls` tags every call with `server`
  (kubernetes/prometheus/github). The console shows it as a colored group chip in
  each tool-call summary; the INFO log prefixes the server, e.g.
  `context tool call: [prometheus] execute_query ...`.

## Flatten Prometheus tool output to value-first text

The pab1it0 prometheus-mcp result is verbose (content-block wrapper, escaped
JSON, noisy labels, a Prometheus UI `links` block) â€” it wastes agent context and
is hard to audit, and range queries can blow up / get truncated. The values were
present (langchain-core 1.x text blocks reach the agent), just buried.

- `src/opsgentic/mcp/context.py`: `_wrap_prometheus_tool` wraps every
  `prometheus` server tool so its output is flattened before it reaches the agent
  and the audit trail. `_flatten_prometheus_result` renders vector results as
  `labels = value` lines and matrix (range) results as per-series
  `last/min/max/n`, dropping `links` and noisy labels (`_PROM_DROP_LABELS`).
  Unexpected shapes (list_metrics, metadata, parse failure) are returned verbatim
  via `_extract_text`, so no data is ever lost. k8s/github tools are untouched.
- `deploy/manifests/agent-skills/sre.md`: instruct the agent to cite concrete
  numeric metric values as evidence, not a Prometheus UI link.
