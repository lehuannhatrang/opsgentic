# 2026-06-22

## Web console (chat, runs/alerts, approvals, skills & config editing)

A small ops console so users can drive OpsGentic from a browser.

### Added
- `console/` — a Node/Express BFF + build-free vanilla-JS SPA (no bundler, deploys as a single `node:20` image). Tabs:
  - **Chat** — trigger `POST /chat` and live-poll the run.
  - **Runs** — list recent runs + detail (status, hypothesis, plan, PR link, **Approve/Reject**).
  - **Skills** — view + edit the `opsgentic-skills` ConfigMap entries.
  - **Config** — edit `opsgentic-config` (agent + MCP env) and set `opsgentic-secrets` keys (masked, write-only).
  Files: `server.js`, `public/index.html`, `package.json`, `Dockerfile`, `.dockerignore`.
- `src/opsgentic/runs.py` `list_recent()` + `runner.list_runs()` + `GET /runs` — recent runs for the console list view (queue mode; `[]` without a DB).
- `deploy/manifests/console/` — `ServiceAccount` + namespaced `Role` (configmaps/secrets get/list/patch; deployments get/patch) + `RoleBinding` + `Deployment` + NodePort `Service` (31090). Added to `kustomization.yaml` (resources + an `images:` entry `opsgentic-console`).

### How it works
- Read/trigger (chat, runs, approve/reject) proxy to the opsgentic API (`OPSGENTIC_API_URL`, default `http://opsgentic.opsgentic.svc:80`).
- Edits (skills/config/keys) go straight to the Kubernetes API via the `opsgentic-console` ServiceAccount (in-cluster token + CA, `https` module, JSON merge-patch), then the console rollout-restarts `opsgentic` + `opsgentic-worker` so the changes take effect (env via envFrom, skills via the cached loader).

### Notes
- Edits are **live** (k8s API), per the chosen design. `kubectl apply -k` re-syncs config/skills from the repo, so persist lasting changes in Git too.
- The console holds namespaced **secrets RW + configmaps RW + deployment restart** — a privileged in-namespace ops tool. It ships as NodePort `31090` with no auth; put it behind an Ingress + authentication before real use.

### Tested
- `py_compile` (runs/runner/main); `node --check server.js`; `kustomize build` — console SA/Role/RoleBinding/Deployment/Service present, image resolves to `opsgentic-console:dev-0.1`.
- Not exercised live (needs the cluster): the k8s patch/restart round-trip and the proxy.

## Console runs table + meaningful PR title/description

### Runs now carry a summary (alert + workload + repo)
- `opsgentic_runs` gains a `summary` jsonb (alert title/source/severity + resolved namespace/service/repo); migrated via `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`. `runs.create` seeds it from the trigger (so even queued/in-flight rows show what's failing); `runner.run_and_track`/`resume_and_track` enrich it with the resolved service/repo after the run. `GET /runs` (list) and `GET /runs/{id}` return it.
- Console SPA: the runs table now shows **Status · Alert · Service (ns/service) · Repo · Src (grafana/chat) · When · PR** (was just thread_id). The `When` column uses **relative time** (now / 10m ago / 2h ago / 3d ago). Rows are clickable; the detail view header shows alert title, source, workload, and repo.

### PR title & description
- `gitops/pr.py`: the PR title is now `"[opsgentic] <alertname> — <namespace>/<service>"` (from the resolved workload + alert) instead of a generic plan summary; the PR body gained an **Alert** section (Source = grafana/chat, alertname, severity, workload) plus a **Change** section. `graph/nodes/action.py` passes `service_ref` as `workload` into both `create_pull_request` and `update_remediation_pr`; `_pr_title`/`_workload_ref` helpers added.

### Tested
- `py_compile` (runs/runner/pr/action); `node --check` on the SPA script + server.js; `kustomize build` OK; PR title/body sanity (no network) -> `[opsgentic] HighPodMemory — payments/payments-api` with an Alert section showing `source: grafana`.
