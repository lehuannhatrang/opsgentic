## MCP connectivity fixes

- **Bug:** `${K8S_MCP_URL}` in `mcp-config/servers.yaml` was not substituted from `.env` — pydantic reads `.env` into Settings but not into `os.environ`, which `os.path.expandvars` relies on. The MCP client then connected to the literal string `${K8S_MCP_URL}` → opaque "TaskGroup" error. Fix: `config.py` calls `python-dotenv` `load_dotenv()` at import (does not override real env vars), so `${VAR}` resolves the same locally and in-cluster. Added `python-dotenv` dependency. Verified: `load_connections()` now resolves the real URL from `.env`.
- **Error surfacing:** `mcp/loader.py` `explain_exception()` flattens ExceptionGroup/TaskGroup errors to the underlying cause; used in `mcp/context.py` and `gitops/cluster.py` warnings so failures show the real error (ConnectError / 404 / transport mismatch) instead of "unhandled errors in a TaskGroup".
- **Diagnostic:** `python -m opsgentic.mcp.diagnose` connects to each configured MCP server, lists tools, and on failure prints the real error and retries the other HTTP transport/path (streamable_http `/mcp` vs sse `/sse`) to reveal which the server speaks.

## MCP output-format + RBAC + context-size fixes (live tuning)

- **RBAC:** added `nodes/proxy`, `nodes/stats`, `nodes/metrics`, `nodes/log` (get) to the read-only ClusterRole so the MCP node-stats/log tools stop returning "Permission denied".
- **Context window blow-up:** the RCA context ReAct agent could accumulate >200K tokens from bulk tools and 400 on the model's context limit. `mcp/context.py` now excludes high-volume tools (`pods_log`, `nodes_log`, `nodes_stats_summary`, `nodes_metrics`) and the prompt asks the agent to be economical.
- **`${VAR}` from `.env`:** `config.py` now calls `python-dotenv` `load_dotenv()` so `${K8S_MCP_URL}` in `servers.yaml` resolves locally (added `python-dotenv` dep).
- **resources_list format:** kubernetes-mcp-server returns a human-readable TABLE wrapped in content blocks (`[{"type":"text","text": ...}]`), not structured objects. `gitops/cluster.py` rewritten: `_text()` unwraps content blocks; `list_resource_refs()` parses the table to (namespace, name); `get_resource()` parses the full object YAML. `resolver._argocd_candidates` now lists refs then fetches each Application's full object. Verified the table parser against real server output.
- Tuning: `MCP_RECURSION_LIMIT` default 25; recommended `LLM_MAX_TOKENS` well below the model context (the value is the per-call output cap, and large values starve the input).

## Auto-patch when the GitOps source path is a directory

- ArgoCD discovery yields `spec.source.path` as a **directory** (e.g. `apps/payments-api`), so auto-patch previously fell back to a proposal markdown. `gitops/pr.py` now: if the resolved path is a file, patch it; if it's a directory, `list_dir` the path via the provider API and locate the manifest whose `kind`/`metadata.name` matches the alerting workload, then patch that file. `action.py` passes `workload=state.service_ref`. Implemented for GitHub/Gitea/GitLab.
- Verified (unit): directory path → picks `deployment.yaml` (the matching Deployment), bumps memory, ignores `service.yaml`/`kustomization.yaml`; file path patches directly; no-patcher falls back to proposal.

Replaces the deterministic single-file heuristic as the primary path: a read-only agent reads the repo and decides which file(s) to change (handles plain manifests, Kustomize, Helm). Writes stay in opsgentic (GitHub App); the agent gets READ tools only.

### Changed (opsgentic)
