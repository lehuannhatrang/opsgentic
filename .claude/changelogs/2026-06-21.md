## Fix: sub-agents inherit the parent's sync PostgresSaver -> NotImplementedError

### Diagnosis
- Live `POST /chat` logged `NotImplementedError:` from both MCP context enrichment and the remediation agent; the graph degraded gracefully (stub context + deterministic patcher) so the PR still opened (only `remediations/*.md` proposed, no manifest edit).
- Root cause (from the real traceback): the transient `create_react_agent(...).ainvoke()` sub-agents run inside `_app.invoke()`, so with `checkpointer=None` they inherit the parent graph's checkpointer via the ambient config. Since M3 that checkpointer is the sync `PostgresSaver`, whose async `aget_tuple` is unimplemented (`langgraph/checkpoint/base` raises `NotImplementedError`); the sub-agent runs async, so `AsyncPregelLoop.__aenter__ -> aget_tuple` raises.
- Explains the symptoms: works locally (no `DATABASE_URL` -> `MemorySaver`, which implements async); fails only on k8s (PostgresSaver); a manual in-pod call works (not nested in `_app.invoke()`, no parent checkpointer to inherit); `resolve_target` works (raw MCP via `gitops/cluster.py`, no inner checkpointed graph). The Postgres deployment added earlier this session is what surfaced it.

### Changed
- `src/opsgentic/mcp/context.py`, `src/opsgentic/gitops/remediator.py` — `create_react_agent(..., checkpointer=False)` so the sub-agents are never checkpointed and never inherit the parent saver.
- Both files — `logger.warning(..., exc_info=True)` on the agent-failure paths so future failures log the real traceback instead of a flattened one-liner.

### Tested
- `py_compile` OK. Reproduced locally: a parent graph compiled with a sync-only saver + an inner `create_react_agent().ainvoke()` -> `checkpointer=None` raises `NotImplementedError`; `checkpointer=False` -> OK (with both `asyncio.run` and a forced SelectorEventLoop, confirming the event loop is irrelevant).
- Pending live confirmation: rebuild, redeploy, `POST /chat` -> `context_data.source=mcp` and agentic (multi-file) remediation.

### Note
- A `run_async`/`asyncutil.py` (force CPython SelectorEventLoop) detour was added then reverted: it was based on a wrong uvloop hypothesis. In-pod tests proved the agent paths run fine on uvloop; the loop type was never the issue.
