## Fix: sub-agents inherit the parent's sync PostgresSaver -> NotImplementedError

### Diagnosis
- Live `POST /chat` logged `NotImplementedError:` from both MCP context enrichment and the remediation agent; the graph degraded gracefully (stub context + deterministic patcher) so the PR still opened (only `remediations/*.md` proposed, no manifest edit).
- Root cause (from the real traceback): the transient `create_react_agent(...).ainvoke()` sub-agents run inside `_app.invoke()`, so with `checkpointer=None` they inherit the parent graph's checkpointer via the ambient config. Since M3 that checkpointer is the sync `PostgresSaver`, whose async `aget_tuple` is unimplemented (`langgraph/checkpoint/base` raises `NotImplementedError`); the sub-agent runs async, so `AsyncPregelLoop.__aenter__ -> aget_tuple` raises.
- Explains the symptoms: works locally (no `DATABASE_URL` -> `MemorySaver`, which implements async); fails only on k8s (PostgresSaver); a manual in-pod call works (not nested in `_app.invoke()`, no parent checkpointer to inherit); `resolve_target` works (raw MCP via `gitops/cluster.py`, no inner checkpointed graph). The Postgres deployment added earlier this session is what surfaced it.

### Changed
- `src/opsgentic/mcp/context.py`, `src/opsgentic/gitops/remediator.py` â€” `create_react_agent(..., checkpointer=False)` so the sub-agents are never checkpointed and never inherit the parent saver.
- Both files â€” `logger.warning(..., exc_info=True)` on the agent-failure paths so future failures log the real traceback instead of a flattened one-liner.

### Tested
- `py_compile` OK. Reproduced locally: a parent graph compiled with a sync-only saver + an inner `create_react_agent().ainvoke()` -> `checkpointer=None` raises `NotImplementedError`; `checkpointer=False` -> OK (with both `asyncio.run` and a forced SelectorEventLoop, confirming the event loop is irrelevant).
- Pending live confirmation: rebuild, redeploy, `POST /chat` -> `context_data.source=mcp` and agentic (multi-file) remediation.

### Note
- A `run_async`/`asyncutil.py` (force CPython SelectorEventLoop) detour was added then reverted: it was based on a wrong uvloop hypothesis. In-pod tests proved the agent paths run fine on uvloop; the loop type was never the issue.

## Fix: intermittent remediation (manifest edit vs proposal-only PR)

### Diagnosis
- The remediation PR sometimes edits `deployment.yaml`, sometimes only adds `remediations/<key>.md` â€” same alert, different runs. `action_node` calls `generate_edits` and passes `edits` to `create_pull_request`; `_compute_changes` is binary (edits -> patch the file; None/empty -> proposal markdown), with no deterministic patcher fallback. The outcome hinges entirely on the ReAct agent emitting edits.
- In-pod measurement on a grafana-derived state (read-only, no PR): `service_ref`/`gitops_target` resolve correctly (`payments-api`, `apps/payments-api`, argocd), but `_run` returned no edits 0/5 with no exceptions â€” the agent reads the repo (github-mcp 200s) and the LLM responds, yet never calls the custom `propose_edit` tool. qwen3 can call tools (it calls github-mcp) but won't reliably emit the custom edit tool; it ends in prose -> `None` -> proposal-only PR.

### Changed
- `src/opsgentic/gitops/remediator.py` â€” replaced the voluntary `propose_edit` tool with `create_react_agent(..., response_format=_RemediationEdits)`. After the read loop, langgraph forces a final structured answer via `with_structured_output()` (a constrained, single-schema call) â€” much more reliable than hoping the model picks a custom tool mid-ReAct. Edits are read from `result["structured_response"]` and mapped to the existing `[{path, ops:[{yaml_path, value}]}]` format.
- Dedup ops per (file, yaml_path), last value wins â€” structured output from qwen3 sometimes repeats the same field many times (observed: one field x10); collapsing avoids a degenerate diff.
- `generate_edits` retries up to `_MAX_ATTEMPTS` (raised 3 -> 6; first attempt with edits wins). Each attempt is read-only. Per-attempt logging (with `exc_info`) distinguishes an errored attempt from a no-edits attempt.
- `_SYSTEM` steers to `resources.requests/limits` over container args for resource pressure, requires at least one edit, and forbids duplicate fields.

### Tested
- `py_compile` OK; `response_format` accepted by the installed `create_react_agent`, output key `structured_response` (langgraph 1.2.6 runs a dedicated `with_structured_output().ainvoke()` after the loop). Dedup verified to collapse the observed 10x-duplicate output to one op.
- In-pod measurement, per-attempt success: 0/5 (voluntary `propose_edit` tool) -> 2/5 (structured output) -> 5/5 (structured output + prompt steer + dedup). The 5/5 run produced an identical, clean, correct edit each time: `apps/payments-api/deployment.yaml` `resources.limits.memory=128Mi`. Deterministic full-file fallback not needed.

## Agent skills: externalize prompts into editable markdown

### Added
- `agent-skills/{sre,gitops,validation,code}.md` â€” editable instruction library. Each file has YAML frontmatter (`name`, `description`, `agents: [...]`) plus a markdown body. A skill is wired to agents via its `agents` field, so adding/rewiring a skill is a pure markdown edit.
- `src/opsgentic/agent_skills.py` â€” loader: parses frontmatter + body, caches (`lru_cache`); `render(agent, default)` composes the bodies of all skills targeting that agent into one system-prompt block, and returns `default` when the library is missing or no skill matches (agents never run promptless).
- `config.py` `skills_path` (default `agent-skills`); `Dockerfile` copies `agent-skills/` into the image.

### Changed
- RCA (`graph/nodes/rca.py`), MCP context enrichment (`mcp/context.py`), remediation (`gitops/remediator.py`), and resolver tiebreak (`gitops/resolver.py`) build their system prompt from `render(<agent>, _FALLBACK)` instead of a hardcoded `_SYSTEM`. Code-bound mechanics stay in code (the remediation edit-schema/output rules `_OUTPUT_RULES`; the resolver index-only answer); reusable domain knowledge lives in the skills.
- Agent -> skill wiring: rca/context -> sre; remediation -> sre+gitops+validation+code; resolver -> gitops.

### Tested
- `py_compile` OK; loader composes the expected skills per agent (rca/context=[sre], remediation=[code,gitops,sre,validation], resolver=[gitops]); missing-library fallback returns the built-in prompt without crashing.

### Note
- Skills are read once and cached, so edits take effect on process restart (and, for the image, after a rebuild). For live edits without rebuild, `agent-skills/` could later be mounted from a ConfigMap.

## Re-fire convergence: stop churning the PR with divergent commits

### Problem
- A re-fired alert re-ran `generate_edits` (non-deterministic) and `create_pull_request` applied the new edit to BASE every time. Since the LLM output varied (128Mi vs 256Mi vs args), the branch content differed each run -> a new divergent commit per re-fire. Nothing considered what the open PR already proposed.

### Changed (policy: re-evaluate; commit only if insufficient; else comment)
- `graph/nodes/action.py` â€” on re-fire, if an open remediation PR exists (`find_open_remediation_pr`), fetch its proposed diff (base...branch) and call `reassess_edits` to judge sufficiency: sufficient -> comment via `update_remediation_pr`; insufficient -> commit an incremental edit relative to the branch. First fire keeps the `generate_edits` + `create_pull_request` path.
- `gitops/remediator.py` `reassess_edits(state, proposed_diff)` â€” a single constrained structured LLM call (no repo navigation): returns `(sufficient, reason, incremental_edits|None)`, biased toward leaving the PR unchanged, defaults to 'sufficient' on any error so a re-fire never churns; dedups ops per (file, yaml_path).
- `gitops/pr.py` (GitHub) â€” `find_open_remediation_pr`, `proposed_diff` (compare base...branch), `update_remediation_pr` (apply edits relative to the branch; if nothing changes, post one de-duplicated issue comment). Non-GitHub providers return None from `find_open_remediation_pr` and keep the existing `create_pull_request` dedup behavior.

### Tested
- `py_compile` OK; local logic test with a fake structured LLM: `reassess_edits` returns sufficient->no edits and insufficient->deduped incremental edits; `action_node` routes first-fire (generate+create) vs re-fire (sufficient=comment, insufficient=incremental commit) correctly.
- Not exercised live (needs the cluster + GitHub App); the compare/comment/commit calls are guarded.
